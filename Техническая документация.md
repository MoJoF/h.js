# Архитектура библиотеки `h.js`

## Общая идея

Основная цель библиотеки — предоставить максимально простой способ создавать реактивный интерфейс без Virtual DOM, JSX и шаблонизаторов.

Основные принципы:

* минимальный размер;
* работа только с настоящим DOM;
* реактивность через сигналы;
* отсутствие диффинга Virtual DOM;
* каждая реактивная часть обновляется независимо от остальных.

---

# Архитектура

Библиотека построена из нескольких уровней.

```text
signal()
      │
      ▼
bind()
bindComputed()
      │
      ▼
processReactive()
      │
      ▼
processCSS()
processText()
processAttrs()
processChildren()
...
      │
      ▼
processors
      │
      ▼
h()
```

Каждый уровень знает только о своей ответственности.

---

# h()

Функция `h()` отвечает исключительно за создание DOM-элемента и распределение свойств между специализированными обработчиками.

Она не знает:

* как работают сигналы;
* как обновляется текст;
* как работают стили;
* как происходит условный рендеринг;
* как рендерятся списки.

Она лишь вызывает нужный обработчик.

```text
h()

↓

создать элемент

↓

перебрать props

↓

найти processor

↓

вызвать processor
```

Благодаря этому функция остаётся небольшой и легко расширяется.

---

# Processors

Каждое свойство элемента обрабатывается собственной функцией.

Например:

* processText()
* processCSS()
* processEvents()
* processAttrs()
* processDataset()
* processChildren()

Добавление нового свойства требует только:

1. написать новую функцию;
2. зарегистрировать её в объекте `processors`.

Код `h()` при этом менять не требуется.

---

# processReactive()

Это центральная функция всей реактивности.

Именно она определяет, что было передано:

* обычное значение;
* Signal;
* вычисляемое значение (computed).

Схема работы:

```text
value

↓

processReactive()

├── function → bindComputed()

├── signal → bind()

└── static → setter()
```

Это позволяет всем обработчикам работать одинаково.

Например:

```js
text: "Hello"

text: titleSignal

text: () => user.name
```

Для `processText()` все три варианта выглядят одинаково.

---

# bind()

Отвечает за подписку на один Signal.

При изменении значения вызывает setter.

Используется тогда, когда свойство напрямую связано с одним сигналом.

---

# bindComputed()

Отвечает за вычисляемые значения.

Во время вычисления включается сбор зависимостей.

Все сигналы, к которым произошло обращение, автоматически подписывают вычисление на себя.

Это позволяет писать:

```js
text: () => `${first.value} ${last.value}`
```

без указания зависимостей вручную.

---

# children

Самая сложная часть библиотеки.

Поддерживает три режима.

## 1. Статические элементы

```js
children: [
    ["div", {...}],
    ["button", {...}]
]
```

Создаются один раз.

---

## 2. Условный рендеринг

```js
children: [
    () => count.value > 5
        ? ["h1", { text: "Много" }]
        : ["h1", { text: "Мало" }]
]
```

Создаётся placeholder.

Во время первого рендера функция выполняется.

Все использованные сигналы автоматически становятся зависимостями.

При изменении любого из них:

* создаётся новый DOM-узел;
* предыдущий заменяется через `replaceWith()`.

Обновляется только этот дочерний элемент.

---

## 3. Рендеринг списков

```js
children: [
    h.each(tasks, item => [
        "li",
        { text: item }
    ])
]
```

Сейчас используется полный повторный рендер списка.

Алгоритм:

* удалить старые элементы;
* пройти по массиву;
* создать новые элементы;
* вставить их перед placeholder.

В дальнейшем возможно добавление алгоритма с ключами (`key`) для повторного использования существующих DOM-узлов.

---

# Преимущества текущей архитектуры

* отсутствует Virtual DOM;
* нет диффинга деревьев;
* каждый реактивный участок независим;
* небольшие специализированные функции;
* легко добавлять новые свойства;
* код хорошо читается;
* большинство обновлений затрагивает только один DOM-узел.

---

# Возможные улучшения

## 1. Keyed each()

Использовать ключи элементов.

Это позволит:

* не удалять весь список;
* перемещать существующие узлы;
* обновлять только изменившиеся элементы.

---

## 2. Fragment

Добавить возможность возвращать несколько соседних элементов без обёртки.

Например:

```js
children: [
    h.fragment(
        ["h1", {...}],
        ["p", {...}]
    )
]
```

---

## 3. Компоненты

Добавить поддержку пользовательских компонентов.

Например:

```js
function Button(props) {
    return h("button", {
        text: props.text
    })
}
```

После этого можно будет писать:

```js
h(Button, {
    text: "Click"
})
```

---

## 4. Очистка зависимостей Computed

Сейчас вычисляемые функции подписываются на сигналы автоматически.

В будущем стоит добавить очистку старых подписок перед повторным вычислением, чтобы избежать накопления неактуальных зависимостей при изменении логики вычисления.

---

# Итог

Архитектура библиотеки уже построена по модульному принципу.

Каждая часть отвечает только за свою задачу:

* `signal()` — хранение состояния;
* `bind()` — реактивная подписка;
* `bindComputed()` — вычисляемые значения;
* `processReactive()` — определение типа значения;
* `process*()` — обработка отдельных свойств;
* `processors` — маршрутизация;
* `h()` — создание DOM-элемента.

Такое разделение делает библиотеку простой для поддержки, удобной для расширения и позволяет постепенно добавлять новые возможности без переработки существующего кода.
